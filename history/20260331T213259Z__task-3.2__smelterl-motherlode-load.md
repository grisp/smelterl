# Task 3.2 - smelterl_motherlode load and schema checks

## Objective

Implement `smelterl_motherlode` so `smelterl plan` can load motherlode
repositories, parse `.nuggets` and `.nugget` files, merge registry defaults,
and fail clearly on malformed input while keeping the later plan pipeline
stubbed for subsequent tasks.

## Initial Investigation

- Task 3.1 added the `smelterl` OTP skeleton and `plan` command validation, but
  `plan` still exits before any motherlode loading occurs.
- The repository still has no Smelterl motherlode loader module or fixture
  coverage for `.nuggets` / `.nugget` schema validation.

## Key Decisions

- Kept Task 3.2 narrowly scoped to motherlode loading plus the minimum `plan`
  integration needed to expose real loader failures; successful `plan` runs
  still stop at the existing "not implemented yet" boundary.
- Used a concrete motherlode structure with:
  - `nuggets := #{NuggetId => NuggetMap}`
  - `repositories := #{}`
  This is enough for the next planning tasks without blocking on VCS metadata.
- Rejected unsupported registry defaults instead of silently ignoring them.
  The design explicitly allows either ignore-or-reject; rejecting keeps schema
  drift visible and gives earlier feedback.
- Attached the declaring nugget identifier to loaded `config` and `exports`
  entries at load time so config consolidation can consume that metadata
  directly in Task 3.8.
- Normalized SBOM default fields into `{Source, Value}` tuples and resolved
  `license_files` to absolute paths at load time, with registry-relative vs
  nugget-relative resolution preserved by the `registry` / `nugget` source tag.

## Residual Risks

- Repositories without a `.nuggets` registry are currently skipped silently.
  The design calls for a warning, but the current Smelterl skeleton has no
  logging/warning surface yet; adding that should happen with later command
  logging work rather than forcing ad-hoc stdout/stderr output into the loader.
- Motherlode loading does not attach VCS provenance yet; `repositories` remains
  empty until a later task introduces `smelterl_vcs` or equivalent metadata
  extraction.
- Nugget field validation is intentionally partial at this stage: root format,
  field-shape sanity, required `id`, supported registry defaults, config/export
  entry shape, file existence for `license_files`, and duplicate nugget IDs are
  checked now; deeper semantic validation belongs to the later tree/validate
  tasks.

## Planning Refinement

- Added a refinement note to `Task 3.8` that config consolidation should reuse
  the `{Key, Value, DeclaringNugget}` entry shape produced by
  `smelterl_motherlode` instead of recomputing origin metadata.
