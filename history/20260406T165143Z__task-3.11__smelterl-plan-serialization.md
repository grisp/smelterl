## Task

Task 3.11: `smelterl_plan` serialization (`build_plan.term`).

## Objective

Replace the current `smelterl plan` stub with real plan serialization, writing
the full deterministic plan payload to `build_plan.term`, persisting the
normalized plan extra-config map and precomputed manifest seed, and covering
the result with focused roundtrip tests.

## Outcome

Added versioned `build_plan.term` serialization through a new `smelterl_plan`
module and completed the `smelterl plan` command path through manifest-seed
construction and plan-file writing. Successful plan runs now emit a reusable
plan payload instead of stopping at the old not-implemented stub.

## Key Decisions

- Serialized the plan as `{build_plan, <<"1.0">>, Fields}` so later
  `generate` work can validate version compatibility explicitly while reusing
  Alloy's existing term-file conventions.
- Stored target-local motherlode views, consolidated config, precomputed
  defconfig models, ordered auxiliary ids, and normalized root-level
  `extra_config` in the plan so future generate/env tasks do not need to
  re-resolve or re-parse planning inputs.
- Added `smelterl_vcs` and motherlode repository attachment in the same task so
  manifest-seed generation can consume real staged-repository provenance from
  `.alloy_repo_info` or git instead of inventing placeholder repository
  records.
- Kept `build_plan.env` out of scope. Task 3.12 remains the owner of the
  shell-friendly export file; this task only writes the authoritative plan
  term.

## Validation

- Baseline gate: `./scripts/tests/gates/baseline.sh`
- Focused Smelterl CT: `rebar3 as test ct --suite test/smelterl_cmd_plan_SUITE.erl --suite test/smelterl_plan_SUITE.erl`
- Full Smelterl CT: `rebar3 as test ct`
- Dialyzer: `rebar3 dialyzer`

## Planning Refinements

- Added a refinement note to Task 3.12 clarifying that `build_plan.env` should
  use the serialized `auxiliary_ids` ordering and normalized root-level
  `extra_config` payload from `build_plan.term` instead of re-deriving either
  from the unordered target map or raw CLI strings.

## Residual Risks

- `smelterl_cmd_generate` does not exist yet, so plan deserialization is
  covered by dedicated roundtrip tests rather than by a full generate-stage
  command flow.
- Dev/test fallback build provenance currently infers the Smelterl repository
  from the checkout when `priv/build_info.term` is absent; the design's
  dedicated build-time `generate_build_info` path is still the long-term source
  of truth for packaged escripts.
