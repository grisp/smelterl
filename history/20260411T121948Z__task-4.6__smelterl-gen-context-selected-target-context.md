# Task 4.6: `smelterl_gen_context` selected-target context

## Objective

Implement selected-target `alloy_context.sh` generation in `smelterl` so
`smelterl generate` can emit one deterministic context per target from the
serialized plan.

## Outcome

Added `smelterl_gen_context`, the `priv/templates/alloy_context.sh.mustache`
template, and `smelterl generate --output-context` wiring in
`smelterl_cmd_generate`. The generate path now renders selected-target context
files from serialized plan data, including target/product metadata,
consolidated config, target-scoped SDK hook arrays, target-local `sdk_outputs`
metadata, and main-only firmware/embed/filesystem-priority sections.
Follow-up work on the same task also tightened Smelterl workflow guidance so
history files omit routine successful validation-command lists, and extended
generate-command coverage to verify that rendered contexts parse in bash,
source successfully in a controlled environment, and remain `shellcheck`
clean apart from the expected `SC2034` warnings for declarative sourced
arrays.

## Key Decisions

- Kept the context generator in the same render/write shape as the other
  generate-stage modules: `smelterl_cmd_generate` still owns plan loading,
  target selection, file opening, and command-level error reporting, while
  `smelterl_gen_context` only transforms selected-target plan data into
  template data and rendered shell output.
- Preserved the strict main-vs-aux boundary in the generated shell surface:
  auxiliary contexts omit firmware capability arrays, embed lists, and
  filesystem-priority metadata, while main contexts include those sections plus
  sdk-output lookup helpers for later Alloy-side auxiliary-output injection.
- Reused plan-carried capability data for firmware variants, variant-specific
  hook selection, merged firmware parameters, and target-local `sdk_outputs`.
  Non-selectable `firmware_outputs` metadata still comes from the selected
  target motherlode because the current capability payload only carries the
  selectable subset.

## Residual Risks / Follow-ups

- The generated context only defines the static shell contract. The main
  context’s runtime auxiliary-output path variables
  (`ALLOY_SDK_OUTPUT_<AUX_ID>_<OUTPUT_ID>` and unique aliases) are still added
  later by Alloy after auxiliary builds, so full end-to-end verification
  remains a follow-up integration concern.
- The new direct context suite is behavior-focused rather than byte-for-byte
  golden today; the later plan/generate integration task is the right place to
  lock the final rendered shell surface against richer end-to-end samples.
