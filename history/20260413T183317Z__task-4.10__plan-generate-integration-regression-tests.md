# Task 4.10: Plan/generate integration regression tests

## Objective

Add end-to-end Smelterl regression coverage for the plan/generate pipeline
using a realistic main-plus-auxiliary sample, with explicit checks that
generate consumes the serialized plan and does not redo dependency resolution.

## Expected Scope

- Add integration-style Common Tests that exercise `smelterl plan` once and
  `smelterl generate` for both an auxiliary target and the main target from the
  emitted plan.
- Cover at least one assertion that the generate stage keeps working from the
  serialized plan even if the original motherlode changes after planning.
- Reuse or extend existing fixtures/helpers where practical so the coverage is
  realistic without duplicating command setup logic across suites.

## Outcome

Added a new end-to-end Common Test suite that drives `smelterl plan` and both
auxiliary and main `smelterl generate` passes from one on-disk sample
motherlode, including a regression that corrupts the source metadata after
planning and verifies generate still renders from the serialized plan. The new
coverage exposed two auxiliary-target pipeline bugs, which were fixed in the
planner: effective auxiliary trees now attach the shared backbone to the
auxiliary root for topology-driven generation, and plan-stage defconfig model
building now resolves flavors from the actual selected target tree instead of a
disconnected synthetic tree.

## Key Decisions

- Kept the new coverage in a dedicated `smelterl_plan_generate_SUITE` rather
  than expanding command-level unit suites further. The task goal is pipeline
  regression coverage across plan and generate together, and a standalone suite
  keeps the end-to-end fixture readable without distorting command-option
  tests.
- Treated the auxiliary-target failures discovered by the new integration test
  as product bugs to fix in the same task instead of weakening the fixture.
  The design expects auxiliary targets to include the shared
  builder/toolchain/platform/system backbone, so topology-driven outputs and
  defconfig planning must work for realistic auxiliary builds.
- Preserved the existing `smelterl_gen_defconfig:build_model/5` test entry
  point and added tree-aware `build_model/6` wiring for plan-stage callers.
  That keeps focused generator tests stable while letting the planner supply
  the real selected-target tree when auxiliary flavor resolution depends on the
  merged backbone.

## Unexpected Issues And Resolution

- The first integration pass failed during planning because auxiliary defconfig
  model building resolved flavors from a disconnected synthetic tree rooted at
  the auxiliary nugget. Passing the real target tree into plan-stage defconfig
  building fixed the missing shared-category validation path.
- After that fix, auxiliary `Config.in`, `external.mk`, and context generation
  still omitted the shared backbone because the composed auxiliary tree kept
  backbone nodes in the edge map without making them reachable from the
  auxiliary root. Updating `smelterl_tree` to attach the auxiliary root to the
  ordered backbone seed nodes fixed topology-driven generation and warranted a
  matching `smelterl_tree_SUITE` expectation update.
- The integration fixture originally expected auxiliary SDK-time hooks from a
  two-tuple `{HookType, Script}` declaration. Smelterl intentionally treats
  that form as `main`-scoped by default, so the suite was corrected to assert
  an empty auxiliary pre-build hook array instead of misclassifying the
  metadata behavior as a pipeline failure.

## Completion Summary

Smelterl now has end-to-end regression coverage for the full plan/generate
pipeline on a realistic main-plus-auxiliary sample, including proof that

auxiliary defconfig planning behave correctly in the real pipeline rather than
only in isolated generate-stage fixtures.
