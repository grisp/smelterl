# Task 4.3: `smelterl_gen_config_in` render/write

## Outcome

Implemented `smelterl_gen_config_in`, added the shared `Config.in` template
under `priv/templates/`, and wired `smelterl generate --output-config-in`
through the existing plan-loading and selected-target validation path. Main
and auxiliary generate passes can now emit deterministic Buildroot `Config.in`
files from the selected target topology plus the plan-carried extra-config key
set, with `ALLOY_MOTHERLODE` always declared first.

## Key Decisions

- Kept `Config.in` rendering in a dedicated generator module and reused
  `smelterl_template` so template lookup, placeholder expansion, and IO stay
  out of `smelterl_cmd_generate`, matching the generate-stage boundary already
  established by Task 4.2.
- Normalized extra-config keys inside the generator instead of trusting caller
  order, which keeps output deterministic even though the serialized plan
  stores extra-config in a map and only guarantees that `ALLOY_MOTHERLODE`
  exists as an injected reserved key.
- Package discovery accepts both a root-level packages-path `Config.in`
  aggregator and per-package subdirectories containing `Config.in`, with
  alphabetical traversal inside each nugget. That keeps the renderer aligned
  with Buildroot package-tree conventions without forcing one directory shape.
