## Objective

Implement the Smelterl capability-discovery stage that derives main-target firmware capabilities and per-target `sdk_outputs` metadata from the overridden target trees. The result must be deterministic and shaped for later context and manifest generation.

## Key Decisions

- Added a dedicated `smelterl_capabilities` module instead of embedding the
  logic in `smelterl_cmd_plan`, so later plan/generate stages can consume one
  reusable discovery surface.
- Kept capability discovery after overrides and topology recomputation, so the
  stage sees the effective target trees and target-local motherlode views that
  downstream plan consumers will also use.
- Normalized firmware output and SDK output metadata into plan-oriented maps
  using `name`/`description` fields, while preserving the documented
  `display_name` metadata input in nugget files.
- Enforced bootflow coverage during discovery, so `smelterl plan` now fails
  before the remaining stub if a discovered firmware variant does not resolve
  to exactly one participating bootflow.

## Unexpected Issues And Resolution

- `smelterl plan` Common Test fixtures that previously reached the stub without
  any bootflow metadata started failing once capability discovery enforced
  variant coverage. The valid-plan fixture was updated to include a plain
  bootflow so command tests still exercise the intended not-implemented path.
- The design text for context generation references all firmware outputs, while
  the documented Task 3.7 capability payload is centered on selectable outputs.
  This task stayed within the documented Task 3.7 payload and recorded a
  follow-up clarification note for Task 4.6.
