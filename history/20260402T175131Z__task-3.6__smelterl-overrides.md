# Task 3.6: `smelterl_overrides` nugget/config/aux remap

## Objective

Implement Smelterl override application for auxiliary remaps, nugget
replacements, and scoped config overrides in deterministic order, then wire it
into the plan pipeline.

## Outcome

Implemented `smelterl_overrides` and integrated it into the `smelterl plan`
pipeline after target topology generation. The override stage now:

- collects override metadata from the main target in deterministic topology
  order,
- applies auxiliary remaps first and reruns target-set validation,
- applies nugget replacements across affected targets, rewrites nugget-id
  references in `depends_on`, and recomputes impacted topologies,
- emits target-local motherlode views with scoped config overrides applied in
  declaration order.

## Key Decisions

- Nugget replacement validation and application both rewrite nugget-id
  references in `depends_on` before revalidation. Without that rewrite,
  dependents would continue to require the replaced nugget id and target-set
  validation would reject a correct replacement.
- The override stage keeps one rewritten shared motherlode for structural
  replacement effects, then forks target-local motherlode views only for
  per-target config overrides. This keeps later stages deterministic while
  preserving scope-specific config results.
- `smelterl_cmd_plan` still ends in the existing not-implemented stub, but only
  after the override stage succeeds. Override failures are now reported through
  command-level diagnostics.

## Residual Risks

- Auxiliary remaps currently validate and support duplicate-id failure paths,
  but the design still leaves little room for a non-duplicating remap shape.
  If later tasks need richer remap semantics, the data-design contract should
  be revisited before expanding behavior.
