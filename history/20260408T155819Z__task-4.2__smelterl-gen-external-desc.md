# Task 4.2: `smelterl_gen_external_desc` render/write

## Outcome

Implemented `smelterl_gen_external_desc` and wired `smelterl generate
--output-external-desc` through the existing plan-loading and selected-target
validation path. Main and auxiliary generate passes now emit deterministic
`external.desc` files from the plan product id plus the selected target's
motherlode view of the product nugget metadata.

## Key Decisions

- Added a narrow `smelterl_template` module and
  `priv/templates/external.desc.mustache` so the first generate-stage text
  renderer follows the template-driven design instead of hardcoding file
  layout in `smelterl_cmd_generate`.
- Kept path opening in `smelterl_cmd_generate` and passed an already-open IO
  device into `smelterl_gen_external_desc:generate/3`, preserving the design
  boundary where the generator writes to caller-provided output without owning
  command-level plan selection or file-path policy.
- Extended `smelterl_file` with reusable UTF-8 iodata writing rather than
  duplicating file/device write handling in the new renderer or template code.

## Residual Notes

- `smelterl_template` currently implements the flat placeholder/render path
  needed by `external.desc`; later generate tasks should extend that shared
  module rather than adding per-generator template lookup or file-writing code.
- A later `grisp_alloy` submodule-pointer sync is still required to record the
  completed Smelterl commit in the superproject, but no immediate Alloy-owned
  task was created because this change is Smelterl-only.
