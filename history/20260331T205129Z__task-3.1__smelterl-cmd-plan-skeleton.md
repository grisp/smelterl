# Task 3.1 - smelterl_cmd_plan skeleton and option validation

## Objective

Implemented the initial `smelterl/` project skeleton required by the backlog
and added a real `plan` command entry path with strict option validation,
Common Test coverage, and a Dialyzer-clean CLI skeleton.

## Initial Investigation

- The repository had no `smelterl/` checkout or Erlang project at all.
- Existing Erlang code in the repository was limited to standalone escripts
  under `scripts/tools/`.
- `Task 3.1` therefore required creating the minimal `smelterl/` project
  structure before the `plan` command handler could exist.

## Key Decisions

- Created the minimal OTP project layout now rather than trying to fake
  `Task 3.1` inside an unrelated existing escript.
- Kept the scope narrow:
  - entrypoint config loading from application env,
  - two-stage CLI dispatch,
  - `smelterl_command` behaviour,
  - `smelterl_cmd_plan` option specification/help/required-option validation,
  - explicit "not implemented yet" failure after validation succeeds.
- Registered only the `plan` command in application env for now. `generate`
  remains a later task instead of being introduced as a placeholder with
  unstable semantics.
- Tested through a real Erlang subprocess so command output and exit status are
  verified together rather than only asserting direct function returns.
- Kept the first-pass global-option reducer simple enough for Dialyzer to prove
  the control flow without impossible fallback branches.
- Updated workflow/test-harness documentation in the same task so Erlang
  validation expectations are explicit: Common Test must pass and Dialyzer must
  report zero warnings for touched Erlang code.

## Residual Risks

- Successful `plan` invocation still stops with `plan execution not implemented
  yet.` because the actual planning pipeline belongs to later tasks
  (`3.2` through `3.12`).
- The CLI parser currently covers only the option shapes needed for `plan`;
  later commands may require extending it when `generate` lands.

## Planning Refinement

- Added a refinement note to `Task 3.2` that motherlode loading should plug
  into the new `smelterl` / `smelterl_cli` / `smelterl_cmd_plan` skeleton
  instead of reworking command registration or CLI shape again.
