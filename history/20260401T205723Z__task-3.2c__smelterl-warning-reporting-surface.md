# Task 3.2c - Smelterl warning/reporting surface

## Objective

Introduce a proper non-fatal Smelterl warning/reporting surface for
command-visible warnings, and use it for motherlode repositories that are
missing `.nuggets` so the loader no longer continues silently.

## Design Grounding

- Read `AGENTS.md`, `docs/00_OVERVIEW.md`, `docs/01_DATA_DESIGN.md`,
  `docs/02_SMELTERL_DESIGN.md`, `docs/03_ALLOY_DESIGN.md`,
  `docs/WORKFLOW.md`, `docs/PLANNING.md`, and `scripts/tests/README.md`.
- Focused design sections:
  - `docs/02_SMELTERL_DESIGN.md` §4.1 Loading the Motherlode
  - `docs/02_SMELTERL_DESIGN.md` §5.3 Documentation
  - `docs/02_SMELTERL_DESIGN.md` §5.7.5 `smelterl_cmd_plan`
  - `docs/02_SMELTERL_DESIGN.md` §5.7.25 `smelterl_log`

## Initial Investigation

- `smelterl_motherlode` currently skips repositories without `.nuggets`
  silently, even though the design says that case should emit a warning.
- `smelterl_cmd_plan` still writes command errors directly with `io:format/3`;
  the Smelterl source tree does not yet contain the dedicated logging/reporting
  module described in the design.
- Focused validation for this task should cover both warning emission and the
  missing-`.nuggets` motherlode path, in addition to Common Test and Dialyzer.

## Key Decisions

- Added a dedicated `smelterl_log` module rather than sprinkling warning
  `io:format/3` calls through planner modules. This keeps command-visible
  diagnostics on one path and matches the design section that already reserves
  `smelterl_log` for stderr reporting.
- Kept the initial logging surface intentionally small but reusable:
  - `error/2`
  - `warning/2`
  - `info/2`
  - `debug/2`
  with `warning` visible by default and `info`/`debug` gated by an
  application-level `log_level` env value for later tasks.
- Used the new warning surface directly inside `smelterl_motherlode` when a
  repository lacks `.nuggets`, so the non-fatal condition is emitted at the
  point of discovery while the load continues.
- Routed existing `smelterl_cli` and `smelterl_cmd_plan` stderr writes through
  `smelterl_log:error/2` so the new module is the canonical reporting path
  instead of being introduced as a one-off warning helper.
- Added command-level Common Test coverage for both:
  - visible warning emission when one repository is missing `.nuggets`
  - deterministic warning ordering when multiple repositories are missing
    `.nuggets`

## Residual Risks

- The focused Common Test helper suites still create predictable `/tmp`
  directory prefixes, so repeated manual reruns can collide with stale
  directories unless they are cleaned first. This task worked around that in
  validation, but the test helper strategy itself remains unchanged.
- `smelterl_log` now provides the canonical warning/error path, but the wider
  Smelterl tree still has only partial adoption because the current codebase is
  limited to the CLI, `plan`, and motherlode loader skeleton.

## Planning Refinement

- Added a refinement note to `Task 3.3` that future non-fatal planner
  diagnostics should route through `smelterl_log` and be covered with
  deterministic command-visible warning tests rather than silent skips or
  ad-hoc stderr output.
