# Task 4.4: `smelterl_gen_external_mk` render/write

## Outcome

Implemented `smelterl_gen_external_mk`, added the shared `external.mk`
template under `priv/templates/`, extended `smelterl_template` with
section-capable Mustache-style rendering, and wired `smelterl generate
--output-external-mk` through the existing plan-loading and selected-target
validation path. Main and auxiliary generate passes can now emit deterministic
Buildroot `external.mk` files from structured package/include data, with the
final comment, blank-line, and `include` layout owned by the template instead
of preformatted Erlang text blocks.

## Key Decisions

- Kept `external.mk` rendering in a dedicated generator module and reused
  `smelterl_template` so template lookup, section rendering, and IO stay
  out of `smelterl_cmd_generate`, matching the same generate-stage boundary
  used by the existing `external.desc` and `Config.in` renderers.
- Reused the same package-tree discovery contract introduced by Task 4.3:
  accept a root-level aggregator file when present (`external.mk` at the
  packages root) and traverse per-package directories alphabetically.
- Included every regular `.mk` file found inside each package directory in
  deterministic alphabetical order. That keeps the output stable while
  avoiding command-layer assumptions about package file naming beyond the `.mk`
  extension.
- Kept the template engine scope intentionally small for this task: variables,
  raw variables, nested sections, and standalone section lines are supported so
  format ownership can move into templates without adding a new dependency or
  trying to solve the entire Mustache surface in one change.
