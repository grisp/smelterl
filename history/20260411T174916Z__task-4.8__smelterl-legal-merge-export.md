# Task 4.8: `smelterl_legal` merge/export multi-target legal trees

## Objective

Implement Smelterl's generate-stage legal merge/export support so one main
pass can combine multiple parsed Buildroot `legal-info/` trees into a single
deterministic export with preserved per-input README content.

## Outcome

Added merged Buildroot legal export support to `smelterl_legal` and wired the
`smelterl generate` command to emit that export when `--export-legal` is
requested. The exporter now parses each repeated `--buildroot-legal` input
through `parse_legal/1`, validates that all inputs report the same Buildroot
version, reconstructs deterministic merged `manifest.csv` and
`host-manifest.csv` files, copies merged `licenses/` and `host-licenses/`
trees, optionally copies `sources/` and `host-sources/`, preserves each input
README block in one template-rendered top-level `README`, copies one
`buildroot.config` from the main input when present, and writes
`legal-info.sha256` over the exported files.

## Key Decisions

- Kept `parse_legal/1` as the single source of truth for Buildroot package
  rows. The exporter rebuilds merged manifests from the parsed package maps
  instead of reparsing or concatenating raw CSV files, and it reconstructs the
  special `buildroot` host row from `br_version`.
- Preserved README blocks in CLI argument order and derived human-facing
  section labels from the standard target workspace path shape
  `targets/<TARGET_ID>/workspace/legal-info` when present. This keeps the
  common SDK-build path readable (`main`, `auxiliary: <AuxId>`) while still
  handling arbitrary legal-info paths deterministically through `input N`
  fallback labels.
- Failed the merge when the same package key (`name`, `version`) appears with
  incompatible metadata or when repeated legal trees disagree on Buildroot
  version. Legal exports should not silently flatten contradictory datasets.
- Scoped this task to Buildroot legal merge/export only. Alloy-owned legal
  artefacts (`alloy-manifest.csv`, `alloy-licenses/`, `alloy-sources/`) and
  manifest finalization remain follow-up work.

## Unexpected Issues And Resolution

- Running multiple `rebar3` Common Test invocations in parallel races on the
  shared `_build/test` beam output path. Final verification was rerun
  sequentially after the initial targeted compile pass hit that rename race.
- Dialyzer flagged the first `write_readme/3` spec because the implementation
  was called with binary paths. The spec was widened to the repo-standard
  `smelterl:file_path()` type.

## Validation Notes

- Added focused Common Test coverage for merged manifest output, README block
  preservation, `buildroot.config` selection, optional source-tree export, and
  `smelterl generate --export-legal` integration.
- Validated against the local Buildroot legal-info reference tree named in the
  task request to confirm the checksum format and README/manifests align with
  real Buildroot output structure.

## Design Changes

- Updated `smelterl/docs/DESIGN.md` to document the current
  `smelterl_legal:export_legal/3` interface and to codify README section-label
  derivation from the standard target-workspace path with deterministic
  fallback.

## Residual Risks / Follow-ups

- Task 4.9 still needs to consume the merged legal package data during
  generate-stage manifest finalization instead of only writing the on-disk
  export.
- Alloy-owned legal artefacts are still deferred; the current export covers
  merged Buildroot legal data only.
- No immediate `grisp_alloy` follow-up task was created because this change is
  Smelterl-owned and the only later superproject work is a deferred/batched
  submodule-pointer sync.
