# Task 3.4 - smelterl_validate target validation

## Objective

Implement `smelterl_validate` so Smelterl rejects invalid target graphs early,
covering category cardinality, target constraints, and auxiliary-target
restrictions before later planning stages consume canonicalized targets.

## Initial Investigation

- `Task 3.3` now builds the main tree plus an ordered auxiliary target list.
- The planning note for `Task 3.4` requires validation to reject duplicate
  auxiliary ids before any later canonicalization can hide them.
- Baseline validation must pass before implementation starts.

## Key Decisions

- Added a dedicated `smelterl_validate` module instead of folding validation
  into `smelterl_tree` or `smelterl_cmd_plan`, so target validation stays
  isolated and reusable for later override/remap work.
- Split validation into two layers:
  - `validate_tree/2` for one tree's category, dependency, conflict, version,
    and flavor rules
  - `validate_targets/2` for cross-target rules such as duplicate auxiliary
    ids, auxiliary-specific forbidden categories, shared-flavor consistency,
    and hook-scope checks
- Wired `smelterl plan` through the new validator before the existing
  not-implemented boundary so invalid target graphs now fail at command level
  with specific diagnostics instead of reaching later pipeline stubs.
- Implemented version checks directly in Smelterl with support for exact,
  comparison, and pessimistic (`~>`) constraints, keeping the validator
  self-contained while the dependency set is still minimal.
- Extended Common Test coverage in three places:
  - new direct validation suite for each validation family
  - `smelterl_cmd_plan_SUITE` coverage for the validated happy path
  - temporary-directory helpers hardened against stale `/tmp` collisions so
    repeated CT/dialyzer runs stay deterministic without manual cleanup

## Residual Risks

- Version-constraint support is intentionally compact and currently covers the
  operators used by the design examples (`=`, comparison operators, and `~>`),
  but it does not attempt to implement a broader external semver DSL beyond
  those forms.
- Hook validation currently enforces scope semantics but does not yet validate
  on-disk hook script existence; that remains aligned with the broader
  motherlode metadata-validation backlog.

## Planning Refinement

- Added a refinement note to `Task 3.6` that override/remap work must rerun
  full target-set validation, not only per-tree validation, so auxiliary-id,
  shared-flavor, and hook-scope invariants remain enforced after tree changes.
