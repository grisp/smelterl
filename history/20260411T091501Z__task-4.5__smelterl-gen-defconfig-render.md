# Task 4.5: `smelterl_gen_defconfig` generate-stage render

## Outcome

Implemented generate-stage defconfig rendering in `smelterl_gen_defconfig`,
added `priv/templates/defconfig.mustache`, and wired
`smelterl generate --output-defconfig` through `smelterl_cmd_generate`. The
generate path now consumes the precomputed `smelterl:defconfig_model()` from
the serialized plan directly and writes a deterministic merged defconfig
without reparsing fragments, recalculating cumulative keys, or touching
Buildroot metadata.

## Key Decisions

- Kept the generate-stage interface narrow: `smelterl_gen_defconfig` now has
  render-only APIs (`render/1`, `render/2`) alongside the existing plan-stage
  `build_model/5`, so plan logic and generate logic stay separated in one
  module rather than being reimplemented in the command layer.
- Used a dedicated `defconfig.mustache` template and the shared
  `smelterl_template` registry so final layout is template-owned, consistent
  with the newer `external.desc`, `Config.in`, and `external.mk` generators.
- Added both direct renderer coverage and plan-driven command coverage so the
  task proves `--output-defconfig` consumes the stored model rather than
  depending on fragment fixtures or cumulative-key spec loading at generate
  time.

## Residual Risks / Follow-ups

- The generate-stage defconfig template currently renders only key/value output
  sections. If later work needs richer generated comments such as per-nugget
  provenance or override annotations like the design appendix examples, that
  should be added as a deliberate template/model extension rather than rebuilt
  from fragment inputs during generate.
