## Task

Task 3.10: `smelterl_gen_manifest` plan-stage seed build.

## Objective

Implement the deterministic manifest-seed builder described by
`smelterl/docs/DESIGN.md` §4.14.A, add focused tests for repository
deduplication/id stability and manifest-seed shape, and centralize the new
shared manifest/build-info types in `smelterl.erl` without extending scope
into generate-stage manifest finalization or plan serialization.

## Outcome

Added `smelterl_gen_manifest:prepare_seed/7` as the plan-stage manifest seed
builder. The new module prepares product metadata, target architecture,
deduplicated repository entries, nugget repository mappings, nugget manifest
records, auxiliary target summaries, firmware capability summaries, top-level
`sdk_outputs`, and external component seed data in a deterministic in-memory
shape that later plan/generate tasks can reuse.

## Key Decisions

- Kept the implementation strictly at the plan-stage seed layer. The new module
  returns the deterministic manifest model but does not serialize `build_plan`
  or finalize `ALLOY_SDK_MANIFEST`; those remain owned by Tasks 3.11 and 4.9.
- Added canonical `repo_id`, `vcs_info`, `smelterl_build_info`, and
  `manifest_seed` types to `smelterl.erl` so later plan/generate work can use
  one shared manifest/build-info contract instead of repeating local type
  declarations.
- Treated nugget repositories with no attached VCS metadata as repository-less
  for seed purposes rather than inventing placeholder provenance. That keeps
  the seed deterministic and avoids encoding fake repository data before the
  later plan integration work wires in real build-info and motherlode VCS
  attachment.

## Validation

- Baseline gate: `./scripts/tests/gates/baseline.sh`
- Focused Smelterl CT: `rebar3 as test ct --suite test/smelterl_gen_manifest_SUITE`
- Full Smelterl CT: `rebar3 as test ct`
- Dialyzer: `rebar3 dialyzer`

## Planning Refinements

- Added a refinement note to Task 3.11 clarifying that plan serialization
  should consume the precomputed manifest seed from `smelterl_gen_manifest`
  and should wire in real build-info/repository provenance instead of
  synthesizing placeholder repository records.

## Residual Risks

- `smelterl plan` still ends before plan serialization, so the new seed builder
  is validated through direct Common Test coverage rather than through
  end-to-end plan output yet.
- Generate-stage manifest finalization is still pending, so path
  relativization, Buildroot legal-info merge, and integrity hashing are not
  exercised by this task.
