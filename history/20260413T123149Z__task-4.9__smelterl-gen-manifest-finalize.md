# Task 4.9: `smelterl_gen_manifest` generate-stage finalize

## Objective

Implement Smelterl's generate-stage manifest finalization so the main
`generate` pass can turn the plan-carried manifest seed into a complete
`ALLOY_SDK_MANIFEST` with runtime fields, optional merged Buildroot legal
sections, relativized license paths, and integrity metadata.

## Expected Scope

- Finalize the manifest from `manifest_seed` during main-target `generate`.
- Reuse merged Buildroot legal package data from `smelterl_legal`.

## Outcome

Implemented generate-stage manifest finalization in `smelterl_gen_manifest`
and wired `smelterl generate` to write `ALLOY_SDK_MANIFEST` from the
plan-carried seed. The finalizer now injects runtime build-environment fields,
relativizes seed and Buildroot license paths to the manifest location, emits
Buildroot package sections when merged legal data is present, and appends the
standard SHA-256 `integrity` section. `smelterl_legal` now exposes
`collect_legal/3` so manifest generation and legal export share one merged
Buildroot legal-data path instead of reparsing exported CSV output.

## Key Decisions

- Kept manifest finalization in `smelterl_gen_manifest` and limited
  `smelterl_cmd_generate` to orchestration. The command layer now collects
  optional legal data once, passes the merged structure into manifest
  generation, and leaves manifest shaping/integrity hashing to the manifest
  module.
- Added `smelterl_legal:collect_legal/3` instead of teaching
  `smelterl_cmd_generate` to merge legal-info trees itself. This keeps the
  Buildroot legal merge logic in one module and lets the manifest path reuse
  the same merged package set as the legal exporter.
- Normalized Buildroot package `license_files` to absolute paths inside the
  merged legal collector, then relativized them back to the manifest base in
  `smelterl_gen_manifest`. That preserves correct manifest paths both when
  legal info stays in target workspaces and when it is exported to
  `legal-info/`.

## Unexpected Issues And Resolution

- Running targeted `rebar3 as test ct` suites in parallel hit the known shared
  `_build/test` beam rename race called out in Task 4.8. Verification was
  rerun sequentially after the first parallel compile failed for that reason.
- The existing `smelterl_cmd_generate_SUITE` plan fixtures used placeholder
  `manifest_seed` values that became invalid once generate-stage finalization
  started validating the seed structure. The fixtures were updated to carry a
  minimal valid manifest seed instead of weakening validation.
