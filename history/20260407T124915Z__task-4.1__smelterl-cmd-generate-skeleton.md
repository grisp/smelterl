# Task 4.1: `smelterl_cmd_generate` skeleton and option validation

## Objective

Implement the initial `smelterl generate` command handler skeleton with strict
option validation for selected-target generation, including main-only option
enforcement for manifest and legal-export flags.

## Scope

- add the command module entry path for `generate`,
- enforce the documented option matrix around `--auxiliary` and main-only
  options,
- add command-level tests for valid and invalid combinations,
- keep the implementation task local to `smelterl/`; any later
  `grisp_alloy` submodule sync can be deferred.

## Outcome

- Added `smelterl_cmd_generate` and registered it in the application command
  handler map, so `smelterl generate` now has first-class CLI dispatch and
  command-specific help.
- The command now requires `--plan`, loads `build_plan.term`, resolves either
  the main target or one named auxiliary target, and reports missing-plan /
  missing-target failures through command-level diagnostics.
- Main-only option gating is enforced in the command layer for
  `--output-manifest`, `--buildroot-legal`, `--export-legal`, and
  `--include-sources`, and the legal-export dependency rules are validated
  before later render tasks exist.
- Added a dedicated Common Test suite for `generate` and updated the shared
  CLI parser so command-argument errors are prefixed with the actual command
  name instead of the previous hardcoded `plan`.

## Decisions

- Kept Task 4.1 intentionally command-layer only: the handler validates and
  selects targets from the serialized plan but does not start file rendering
  yet, leaving output generation to the dedicated Task 4.2+ modules.
- Kept main-vs-auxiliary option enforcement in `smelterl_cmd_generate` rather
  than deferring it to later generator modules, so later render stages can
  rely on a single validated command contract.
- Recorded the follow-up refinement under Task 4.2 to reuse this plan-loading
  and selected-target validation path instead of rebuilding it in each render
  task.

## Residual Risks

- `smelterl generate` currently stops after option validation, plan loading,
  and target selection; output rendering and file writes remain for Task 4.2+
  and integration coverage for the full generate pipeline remains for Task
  4.10.
- A later `grisp_alloy` follow-up is still required when the superproject is
  ready to record the updated `smelterl/` submodule commit, but that sync can
  remain batched until Alloy-side work resumes.
