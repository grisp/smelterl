# Task 3.4b - Smelterl Appendix B case-depth and maybe readability rule

## Objective

Update Smelterl Appendix B so it explicitly guides use of `maybe` for readable
linear fallible flows, defines a preferred nesting threshold for `case`
expressions, and refactor the current Smelterl Erlang modules that now violate
that rule where the control flow becomes clearer.

## Design Grounding

- Read `AGENTS.md`, `docs/00_OVERVIEW.md`, `docs/01_DATA_DESIGN.md`,
  `docs/02_SMELTERL_DESIGN.md`, `docs/03_ALLOY_DESIGN.md`,
  `docs/WORKFLOW.md`, `docs/PLANNING.md`, and `scripts/tests/README.md`.
- Focused design sections:
  - `docs/02_SMELTERL_DESIGN.md` Appendix B
  - current `smelterl_validate` and `smelterl_cmd_plan` control-flow style

## Initial Investigation

- The current Smelterl Erlang tree does not use the Erlang `maybe` syntax yet.
- `Task 3.4` introduced several linear error-propagation paths where `maybe`
  would improve readability.
- Baseline validation must pass before implementation starts.

## Key Decisions

- Added the readability rule to Appendix B as guidance rather than a rigid
  mechanical transformation:
  - use `maybe` when it materially improves linear fallible flows
  - treat more than three nested `case` expressions as the default refactoring
    threshold
  - prefer helper extraction when branching logic is not naturally linear
- Applied the rule conservatively to the current codebase:
  - refactored the clear success/error ladders in `smelterl_validate`
  - refactored `smelterl_cmd_plan` where `maybe` plus small wrapper helpers
    removed nested control flow without changing the command-visible behavior
- Kept explicit `case` expressions where they still carry the branching logic
  more clearly than a forced `maybe` conversion.

## Residual Risks

- The new Appendix B rule is intentionally readability-driven, so future tasks
  will still need judgment rather than a blind search/replace on `case`
  expressions.
- The refactor touched only the current obvious linear ladders; more complex
  future modules should apply the same rule as they are introduced or revised.

## Planning Refinement

- Added a refinement note to `Task 3.5` so upcoming Smelterl Erlang work
  follows the new Appendix B `maybe` and case-depth guidance instead of
  reintroducing deep nested `case` ladders.
