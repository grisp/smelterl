# Task 3.3 - smelterl_tree main and auxiliary tree construction

## Objective

Implement `smelterl_tree` so Smelterl can build the main target tree, discover
auxiliary targets from `auxiliary_products`, build auxiliary-specific
subtrees, and compose effective auxiliary trees with deterministic dependency
edge ordering and cycle detection.

## Initial Investigation

- The repository currently has motherlode loading, CLI dispatch, and
  warning/reporting support, but no tree-construction module yet.
- The design splits responsibilities deliberately:
  - `smelterl_tree` builds dependency subtrees and detects cycles
  - command-level orchestration owns auxiliary discovery and effective-tree
    composition with the main backbone
- `Task 3.3` therefore needs both a reusable subtree builder and enough target
  orchestration surface to return main + effective auxiliary trees for the next
  validation task.

## Key Decisions

- Added a dedicated `smelterl_tree` module with two layers:
  - `build/2` for one nugget-only dependency subtree with cycle detection
  - `build_targets/2` for plan-stage main-tree build, auxiliary discovery, and
    effective auxiliary-tree composition with the main backbone
- Kept tree construction strict about nugget-ordering inputs while leaving
  later semantic validation to `Task 3.4`:
  - `required`, `optional`, `one_of`, and `any_of` nugget constraints affect
    tree loading
  - `category`, `capability`, and `conflicts_with` constraints do not create
    edges at this stage
  - version/flavor properties are carried forward for later validation rather
    than enforced during subtree loading
- For `one_of`/`any_of` nugget constraints, tree construction loads all
  currently available candidate nuggets in declaration order and fails only if
  none are available. This preserves enough structure for the validation stage
  to enforce exclusivity/cardinality without losing ordering information.
- Preserved direct dependency order exactly as declared in `depends_on` so the
  later topology task can use the tree edges as its deterministic tie-breaker
  instead of re-reading motherlode metadata.
- Kept discovered auxiliary targets as an ordered list, not a map, so duplicate
  `AuxId` declarations are still visible to `Task 3.4` and are not silently
  overwritten before validation can reject them.
- Wired `smelterl_cmd_plan` through the tree-construction step before the
  existing "not implemented yet" boundary so command-level failures now surface
  circular dependencies and missing required nugget dependencies immediately.

## Residual Risks

- Auxiliary-root version and flavor constraints are discovered and preserved,
  but they are not enforced during tree construction; that remains the job of
  `Task 3.4`.
- The Common Test temp-directory strategy still uses predictable `/tmp`
  prefixes, so repeated manual reruns need the same cleanup step already used
  in earlier Smelterl tasks.

## Planning Refinement

- Added a refinement note to `Task 3.4` that duplicate `AuxId` declarations are
  intentionally preserved in the pre-validation auxiliary target list; the
  validator should reject them before any later target-id canonicalization.
